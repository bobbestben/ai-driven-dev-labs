# Backend Architecture Rules

# Packages

- should be split by domain, not layer (we use DDD style)
- Inside folder `greenfield-backend` or `brownfield-backend`, source files live under `src/` with one subfolder per domain (e.g. `src/pet/`, `src/vet/`)

# Backend Coding practices

- as much as possible, functions should be single responsibility. If a function is doing more than one thing, split it into several functions
- for Repository/Service/Router, use method action words such as `find`, `save`, `delete`. Do not use `create` or `remove`

# Refactoring

- When refactoring code, you **MUST** update the existing Vitest tests to ensure they pass and reflect the changes. Never leave tests in a broken state.
- If the refactoring introduces any new behavior, you **MUST** add new tests to cover it.

## Database

- schema should be defined in `src/data.sql`
- when having a data creation script inside `data.sql`, always make sure that data is populated in the right order, so we do not assign null values into columns which are "not null"
- use SQLite-compatible SQL syntax (e.g. `INTEGER PRIMARY KEY AUTOINCREMENT` instead of `BIGINT GENERATED BY DEFAULT AS IDENTITY`)
- the database is SQLite in-memory (`:memory:`); the `data.sql` file is executed once on startup

## For TypeORM entities

- use TypeORM decorators: `@Entity()`, `@PrimaryGeneratedColumn()`, `@Column()`, `@ManyToOne()`, `@OneToOne()` etc.
- properties should be `readonly` — no setters
- there should be a constructor that accepts all non-id fields as optional parameters
- use `@Column({ name: "snake_case_name" })` only when the property name differs from the column name
- do not use bidirectional relationships unless really needed
- keep relationship decorators minimalistic — do not add extra options unless explicitly required

## For Repository classes

- create a plain TypeScript class wrapping `DataSource.getRepository(Entity)`
- accept an optional `DataSource` parameter in the constructor so tests can inject an in-memory instance
- method names: use `find*`, `save`, `delete`

## For Service classes

- create a plain TypeScript class
- accept an optional Repository parameter in the constructor for testability
- never implement a Service class without a Vitest test

## For Router (controller) files

- use Express `Router`
- use URL pattern `/api/v1/ENTITY` (e.g. `/api/v1/pets`)
- file name suffix: `Router.ts` (e.g. `petRouter.ts`)
- add JSDoc `@openapi` comments above each route for Swagger generation

## Vitest tests - what to test

- we should have good coverage of Vitest tests at the service layer
- if business rules are simple, do an integration test (all the way down to an in-memory SQLite database)
- all complex business rules should be tested with a unit test using `vi.mock()`
- only test Repository and Router if there is something interesting to test

## Vitest tests - Implementation

- all test files should follow the convention `<fileName>.test.ts` (e.g. `vetService.test.ts`)
- method names: should use `should` and `shouldNot` (e.g. `should find pet by name`)
- inside test functions, use the `given/when/then` structure with a blank line between each section
- integration tests use a dedicated `createTestDataSource()` helper that creates a fresh in-memory SQLite DataSource and runs `data.sql`
- when generating test data, make sure there is no conflict with the seed data inside `data.sql`
